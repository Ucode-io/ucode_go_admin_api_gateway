# Google Lead Form Ads — гайд для фронтенда

Этот документ описывает **всё, что нужно фронту**, чтобы построить UI интеграции Google Lead Form Ads:
как создать подключение, как показать пользователю «шпаргалку» для настройки в Google Ads,
как собрать маппинг полей, как проверить работу и как показывать понятные сообщения.

Идентификаторы, поля и JSON — на английском. Пояснения — по-русски.

> ⚠️ **Главное отличие от Meta.** Здесь **НЕТ OAuth, НЕТ попапа логина, НЕТ запроса страниц/форм из API.**
> Google сам присылает нам данные лида на вебхук. Подключение настраивает **сам пользователь руками
> в кабинете Google Ads** — мы лишь даём ему две строки (Webhook URL + Key) и принимаем лиды.

---

## 1. Что это вообще такое (модель в голове)

Пользователь крутит рекламу в Google с **лид-формами** (Lead Form Ads). Когда человек заполняет форму,
рождается **лид**. Google **сам отправляет этот лид** на наш вебхук вместе со всеми данными, а мы
кладём его строкой в таблицу проекта по настроенному **маппингу полей**.

Поток в двух мирах:

```
В ucode (наш UI):                         В Google Ads (кабинет пользователя):
─────────────────                         ──────────────────────────────────
1. Создать интеграцию                      4. Открыть свою лид-форму
   (выбрать таблицу + маппинг)                → Lead delivery → Webhook
2. Получить google_key + webhook_url          → вставить Webhook URL + Key
3. Скопировать обе строки  ───────────────►   5. Нажать «Send test data»
                                              6. Сохранить
                          ◄───── лиды летят на /webhook/google ──────
7. Лиды автоматически появляются в таблице проекта
```

Ключевые понятия:

| Термин | Что это |
|---|---|
| `google_key` | **Секрет, который генерим МЫ** при создании интеграции. Пользователь вставляет его в Google Ads. Google возвращает его в каждом лиде — по нему мы и аутентифицируем, и находим проект. |
| `webhook_url` | Публичный URL нашего бэка, куда Google шлёт лиды. Пользователь вставляет его в Google Ads. |
| `column_id` | Код поля Google-формы (напр. `FULL_NAME`, `EMAIL`, `PHONE_NUMBER`) — левая сторона маппинга. |
| `table_slug` | Slug целевой таблицы проекта, куда писать лиды. |
| `form_id` | (опционально) ID конкретной лид-формы в Google Ads. |

> Фронт **не общается с Google напрямую**. Все запросы идут на наш gateway, а лиды Google шлёт на наш вебхук сам.

## 2. Общие правила для всех запросов

### 2.1. База и авторизация

- Все эндпоинты ниже — на нашем gateway, префикс `/v1/google-leads`.
- Авторизация — **те же заголовки, что и для любого другого запроса к gateway**:

| Заголовок | Назначение |
|---|---|
| `Authorization: Bearer <token>` | Токен пользователя платформы |
| `Environment-Id: <uuid>` | Текущее окружение проекта |
| `Resource-Id: <uuid>` | Как обычно в проекте |

`project_id` и `user_id` сервер достаёт сам из токена — **их слать не нужно**.

Публичный эндпоинт `/webhook/google` фронт **не вызывает** — его дёргает сам Google.

### 2.2. Единый формат ответа (envelope)

Любой ответ gateway завёрнут в один конверт:

```json
{
  "status": "OK",
  "description": "The request has succeeded",
  "data": { /* полезная нагрузка ИЛИ текст ошибки */ },
  "custom_message": null
}
```

- Полезные данные эндпоинта — всегда внутри **`data`**.
- При ошибке в `data` лежит **строка с текстом ошибки**, а `status` — машинный код.

### 2.3. Коды статусов

| HTTP | `status` | Когда |
|---|---|---|
| 200 | `OK` | Успех |
| 400 | `BAD_REQUEST` | Невалидное тело / бизнес-ошибка |
| 400 | `INVALID_ARGUMENT` | Не передан/кривой обязательный параметр |
| 500 | `GRPC_ERROR` | Ошибка внутреннего сервиса |
| 500 | `INTERNAL_SERVER_ERROR` | Непредвиденная ошибка |
| 401 | `UNAUTHORIZED` | Проблема с авторизацией платформы |

Правило для UI: если `status !== "OK"` — показываем ошибку, текст берём из `data` (это строка).

## 3. Полный поток подключения (для UX)

В отличие от Meta здесь нет OAuth-редиректа. Поток линейный и простой:

1. Пользователь выбирает **таблицу** проекта, куда писать лиды.
2. Строит **маппинг**: какое поле Google-формы → в какую колонку таблицы.
3. Жмёт **«Создать подключение»** → `POST /v1/google-leads`.
4. Мы возвращаем **`google_key` + `webhook_url`**. Показываем их пользователю с кнопками «Скопировать».
5. Пользователь идёт в **Google Ads**, вставляет обе строки в настройки лид-формы (см. шпаргалку §8), жмёт **«Send test data»**.
6. Тестовый лид прилетает к нам и **появляется строкой в таблице** — пользователь видит, что всё работает.
7. Готово. Реальные лиды падают автоматически.

> Поскольку правки маппинга не требуют переподключения, пользователь может менять соответствие полей в любой
> момент через `PUT /mapping/:id` — `google_key` и `webhook_url` остаются те же.

    { "column_id": "WORK_EMAIL",    "label": "Work email" },

## 5. Создание подключения (Create)

**`POST /v1/google-leads`**

Генерирует `google_key`, сохраняет маппинг и возвращает ключ + webhook URL для вставки в Google Ads.

Тело:

```json
{
  "form_id": "123456789",
  "form_name": "Заявка на консультацию",
  "table_slug": "leads",
  "fields": [
    { "lead_column": "FULL_NAME",    "table_field": "name",  "required": true },
    { "lead_column": "PHONE_NUMBER", "table_field": "phone", "required": true },
    { "lead_column": "EMAIL",        "table_field": "email", "required": false },
    { "lead_column": "my_custom_q1", "table_field": "note",  "required": false }
  ]
}
```

| Поле | Обязательно | Смысл |
|---|---|---|
| `form_id` | нет | ID лид-формы Google. Если задан — принимаем **только** лиды этой формы; пусто — любые лиды с этим ключом |
| `form_name` | нет | Для отображения (станет именем подключения) |
| `table_slug` | **да** | Slug целевой таблицы проекта |
| `fields[].lead_column` | **да** | `column_id` Google (стандартный из `/columns` или кастомный) |
| `fields[].table_field` | **да** | Колонка в таблице проекта |
| `fields[].required` | нет | Если `true` и значение пустое — лид **не запишется** |

`data`:

```json
{
  "resource_id": "0e4b...uuid",
  "google_key": "9f2c1ab34de5...48символов-hex",
  "webhook_url": "https://api.ucode.io/webhook/google"
}
```

- `resource_id` — ID подключения (для маппинга и отключения).
- `google_key` — **показать пользователю с кнопкой «Скопировать»**.
- `webhook_url` — то же. ⚠️ Если пришёл пустым — на бэке не задан env `GOOGLE_LEADS_WEBHOOK_URL`; покажите
  заглушку «Webhook URL появится после настройки сервера, обратитесь к администратору».

Ошибки:

| status | `data` | Показать |
|---|---|---|
| `BAD_REQUEST` | ошибка валидации (нет `table_slug` / пустые `fields`) | Подсветить незаполненные поля |
| `GRPC_ERROR` | текст | «Не удалось создать подключение. Попробуйте позже.» |

## 6. Список подключений (Integration)

**`GET /v1/google-leads/integration`**

Все Google-подключения текущего проекта/окружения со статусом, ключом и маппингом.

`data`:

```json
{
  "connected": true,
  "integrations": [
    {
      "connected": true,
      "resource_id": "0e4b...uuid",
      "google_key": "9f2c1ab34de5...",
      "webhook_url": "https://api.ucode.io/webhook/google",
      "form_id": "123456789",
      "form_name": "Заявка на консультацию",
      "table_slug": "leads",
      "status": "active",
      "connected_at": "2026-06-30T08:30:00Z",
      "fields": [
        { "lead_column": "FULL_NAME",    "table_field": "name",  "required": true },
        { "lead_column": "PHONE_NUMBER", "table_field": "phone", "required": true }
      ]
    }
  ]
}
```

| Поле | Смысл |
|---|---|
| `connected` (верхний) | Есть ли хоть одно подключение |
| `resource_id` | ID подключения (для mapping/disconnect) |
| `google_key` / `webhook_url` | Чтобы пользователь мог снова скопировать в Google Ads |
| `form_id` / `form_name` | Какая форма (form_id может быть пустым = любая) |
| `table_slug` | Целевая таблица |
| `status` | `active` |
| `connected_at` | Когда создали (RFC3339) |
| `fields` | Текущий маппинг |

Это основной источник для экрана «Мои подключения»: карточка на каждый `integrations[]`,
кнопки «Показать ключ», «Маппинг», «Отключить».

## 7. Обновление маппинга (Mapping)

**`PUT /v1/google-leads/mapping/:id`**

`:id` — `resource_id` подключения. Обновляет таблицу/маппинг, **`google_key` и `webhook_url` не меняются**
(переподключать в Google Ads не нужно).

Тело (идентично create, кроме того, что ключ не генерируется):

```json
{
  "form_id": "123456789",
  "form_name": "Заявка на консультацию",
  "table_slug": "leads",
  "fields": [
    { "lead_column": "FULL_NAME",    "table_field": "name",  "required": true },
    { "lead_column": "PHONE_NUMBER", "table_field": "phone", "required": true },
    { "lead_column": "CITY",         "table_field": "city",  "required": false }
  ]
}
```

`data`:

```json
{ "resource_id": "0e4b...uuid" }
```

Ошибки:

| status | `data` | Показать |
|---|---|---|
| `INVALID_ARGUMENT` | `resource id is required` | Тех. ошибка |
| `BAD_REQUEST` | ошибка валидации | Подсветить незаполненные поля |
| `BAD_REQUEST` | `google leads integration is not configured for this resource` | Подключение не найдено/битое — обновить список |
| `GRPC_ERROR` | текст | «Не удалось сохранить маппинг. Попробуйте позже.» |

### Как собрать UI маппинга

1. Берём `table_slug` → тянем поля таблицы проекта (вашим существующим API схемы таблиц).
2. Слева — выбор `column_id`: дропдаун из `GET /columns` (`label`) + опция «Свой column_id» (текстовый ввод для кастомных вопросов).
3. Справа — селект полей таблицы (`table_field`).
4. Чекбокс `required` на строку.
5. Кнопка «+ Добавить поле» добавляет строку.
6. По «Сохранить» — собираем `fields[]` и шлём create или `PUT /mapping/:id`.

## 8. 🧾 Шпаргалка для пользователя (настройка в Google Ads)

Это текст/визард, который фронт показывает рядом с `google_key` и `webhook_url`. Цель — чтобы пользователь
**без поддержки** сам подключил вебхук в Google Ads.

> **Как подключить лиды Google к ucode**
>
> 1. Зайдите в **Google Ads** → ваша кампания с **лид-формой** (Lead form asset).
> 2. Откройте форму на редактирование → раздел **«Lead delivery / Доставка лидов»**.
> 3. Включите **«Webhook integration / Интеграция через вебхук»**.
> 4. Вставьте:
>    - **Webhook URL** → `<webhook_url>`  *(кнопка «Скопировать»)*
>    - **Key** → `<google_key>`  *(кнопка «Скопировать»)*
> 5. Нажмите **«Send test data / Отправить тестовые данные»** — придёт тестовый лид.
> 6. Нажмите **«Сохранить»**.
> 7. Вернитесь сюда и нажмите **«Проверить подключение»** (см. §9) — тестовый лид должен появиться в таблице.

Важные подсказки в UI:
- Кнопки **«Скопировать»** на обе строки — обязательно (длинный hex руками не перепишешь).
- Подпись «Ключ — это секрет, не публикуйте его».
- Ссылка «Где это в Google Ads?» со скриншотом/гифкой раздела Lead delivery.

## 9. Проверка работы («Проверить подключение»)

После того как пользователь нажал в Google Ads **«Send test data»**, тестовый лид (`is_test=true`)
прилетает к нам и **записывается в таблицу так же, как боевой** (по решению продукта тестовые лиды пишем).

Как фронту показать проверку (вариантов два, можно оба):

1. **По таблице (рекомендуется):** после нажатия «Проверить подключение» опросите таблицу `table_slug`
   вашим обычным data-API (последние записи) — если появилась новая строка с тестовыми значениями, показываем
   ✅ «Подключение работает, лид получен».
2. **По статусу подключения:** `GET /integration` → если `status: "active"`, подключение существует
   (но факт прихода лида это не доказывает — поэтому пункт 1 надёжнее).

UX-подсказка: дайте кнопку «Проверить ещё раз» и небольшой поллинг (например, 5 попыток по 2 сек),
т.к. лид обрабатывается асинхронно (придёт за пару секунд).

> Тестовые строки потом можно удалить из таблицы вручную — об этом стоит мягко предупредить.

## 10. Отключение (Disconnect)

**`DELETE /v1/google-leads/integration/:id`**

`:id` — `resource_id` из `/integration` или из ответа create. Удаляет подключение.

`data`:

```json
{ "resource_id": "0e4b...uuid" }
```

Ошибки:

| status | `data` | Показать |
|---|---|---|
| `INVALID_ARGUMENT` | `resource id is required` | Тех. ошибка |
| `GRPC_ERROR` | текст | «Не удалось отключить. Попробуйте позже.» |

UI: подтверждение («Новые лиды перестанут поступать. В Google Ads не забудьте убрать вебхук»),
затем рефетч `/integration`.

> ⚠️ Важно сообщить: удаление в ucode **не убирает** вебхук в Google Ads. Лиды просто перестанут находить
> проект (бэк ответит 200 и проигнорирует). Чтобы Google не слал зря — пусть уберёт вебхук и у себя.

## 12. Справочник эндпоинтов

| Метод | Путь | Назначение |
|---|---|---|
| GET | `/v1/google-leads/columns` | Стандартные `column_id` для маппинга |
| POST | `/v1/google-leads` | Создать подключение → вернуть `google_key` + `webhook_url` |
| GET | `/v1/google-leads/integration` | Список подключений + маппинги + ключи |
| PUT | `/v1/google-leads/mapping/:id` | Обновить маппинг (ключ не меняется) |
| DELETE | `/v1/google-leads/integration/:id` | Удалить подключение |

Публичный (Google → backend, фронт не трогает): `POST /webhook/google`.

## 13. Приложение: как лид попадает в таблицу (контекст)

Фронт здесь ничего не делает — для понимания общей картины:

1. Человек заполняет лид-форму Google.
2. Google шлёт на наш `POST /webhook/google` **все данные лида** + `google_key` + `form_id`.
3. Backend по `google_key` находит подключение(я), проверяет ключ, при заданном `form_id` сверяет форму.
4. Раскладывает `user_column_data` (`column_id → value`) по маппингу в колонки таблицы.
5. Пишет строку в `table_slug`. Дубликаты защищены детерминированным `guid` от `lead_id`.

То есть после create + вставки ключа в Google Ads лиды появляются в таблице автоматически, без участия UI.
Отличие от Meta: **нет второго запроса за данными** (Google присылает всё сразу) и **нет OAuth-токенов**.
