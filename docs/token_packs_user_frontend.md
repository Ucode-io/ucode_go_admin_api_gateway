# Token Packs — документация для фронтенда (сторона пользователя)

Этот документ описывает всё, что нужно фронтенду, чтобы реализовать пользовательскую часть «токен-пакетов»: что это, как устроена логика, какие эндпоинты дёргать, какие тела запросов/ответов, какие ошибки и как построить хороший UX.

Админская часть (создание/редактирование/удаление пакетов) — в отдельном файле `token_packs_admin_frontend.md`.

---

## 1. Суть фичи (обязательно к прочтению)

У каждого проекта есть тариф (fare) с лимитом AI-токенов: **дневной** и **месячный**. Когда хоть один из этих лимитов исчерпан, обычно AI-работа остановилась бы. **Токен-пакет** — это докупаемый запас токенов, который автоматически подхватывается, когда лимит тарифа достигнут, чтобы работа не вставала.

Ключевые свойства (важно для правильного UI):

- **Автоматическое переключение.** Пользователь ничего не включает руками. Бэкенд сам тратит токены так: сначала тариф, и **как только достигнут дневной ИЛИ месячный лимит** — автоматически начинает тратить пакет. Когда лимит тарифа снова доступен (дневной сбрасывается в полночь UTC, месячный — в начале месяца) — бэкенд так же автоматически возвращается на тариф, а пакет не трогается.
- **Пакет не имеет дневного/месячного лимита.** Купленные токены можно потратить хоть все за один день, хоть растянуть на год.
- **Пакет не сгорает.** Купил один раз — лежит, пока не израсходуется.
- **Баланс пакета — на уровне компании (company), а не проекта.** Все проекты компании тратят из одного общего пула. Но *оплата* покупки списывается с баланса конкретного проекта (`project.balance`), под которым пользователь авторизован.

> Поэтому в UI правильно говорить: «у компании осталось N токенов в пакете», а покупку оформлять от имени текущего проекта.

## 2. Аутентификация и формат ответа

Все эндпоинты находятся под `/v1` и требуют те же заголовки авторизации, что и остальные вызовы gateway (`Authorization` + `project_id` в заголовках). Бэкенд сам по проекту определяет компанию и пользователя — фронту **не нужно** передавать `company_id`/`user_id` в теле.

Каждый ответ обёрнут в стандартный конверт:

```json
{
  "status": "OK",
  "description": "The request has succeeded",
  "data": { /* полезная нагрузка — описана ниже для каждого эндпоинта */ },
  "custom_message": null
}
```

Полезные данные всегда лежат в `data`. Поля с нулевыми/пустыми значениями могут отсутствовать в JSON (omitempty), поэтому при чтении считайте отсутствующее поле равным `0`/`""`/`false`.

## 3. Эндпоинты пользовательской стороны

### 3.1. Список доступных пакетов

```
GET /v1/token-pack?only_active=true&product_type=ugen
```

Query-параметры (все опциональны):

| параметр       | тип     | описание |
|----------------|---------|----------|
| `only_active`  | bool    | `true` — показать только активные пакеты (используйте это в магазине). Если не передать — вернутся и неактивные. |
| `product_type` | string  | фильтр по типу продукта: `ugen` или `ucode`. Обычно `ugen`. |

`data`:

```json
{
  "token_packs": [
    {
      "id": "b3f1...uuid",
      "name": "1M tokens",
      "token_amount": 1000000,
      "price": 50000,
      "currency_id": "0803582e-29d6-42fe-86ac-b4287b8fa929",
      "product_type": "ugen",
      "is_active": true,
      "created_at": "2026-06-29T10:00:00Z",
      "updated_at": "2026-06-29T10:00:00Z",
      "currency": { "id": "...", "symbol": "сум", "name": "Uzbek Som", "code": "UZS" }
    }
  ],
  "count": 1
}
```

Поля пакета:

- `token_amount` — сколько токенов даёт пакет.
- `price` + `currency` — цена в указанной валюте (может быть USD; бэкенд при покупке сам пересчитает в UZS по курсу ЦБ).
- `is_active` — для пользователя показывайте только `true`.

**UI:** карточки/таблица пакетов с названием, количеством токенов и ценой. Отсортированы бэкендом по возрастанию `token_amount`.

---

### 3.2. Текущий баланс пакета компании

```
GET /v1/token-pack/balance
```

Без тела. Компания определяется по проекту из авторизации.

`data`:

```json
{ "company_id": "…", "remaining_tokens": 740000 }
```

`remaining_tokens` — сколько токенов осталось в общем пуле компании. Если пакет ни разу не покупали — вернётся `0`.

---

### 3.3. Покупка пакета

```
POST /v1/token-pack/purchase
Content-Type: application/json

{ "pack_id": "b3f1...uuid" }
```

В теле — **только** `pack_id`. `company_id`, `project_id`, `user_id` бэкенд берёт из контекста авторизации.

Что происходит на бэке атомарно: списывает цену пакета с `project.balance` (USD→UZS по курсу при необходимости), создаёт транзакцию типа `token_pack`, начисляет токены в пул компании.

`data` при успехе (HTTP 200):

```json
{
  "company_id": "…",
  "tokens_added": 1000000,
  "remaining_tokens": 1740000,
  "charged_amount": 50000,
  "transaction_id": "…"
}
```

- `tokens_added` — сколько токенов добавилось.
- `remaining_tokens` — новый общий баланс пула после покупки (можно сразу обновить им UI).
- `charged_amount` — сколько фактически списано **в UZS**.
- `transaction_id` — id созданной транзакции (для истории/чека).

#### Ошибки покупки (важно — у каждого случая свой HTTP-код)

| HTTP | когда | что показать |
|------|-------|--------------|
| `402 PAYMENT_REQUIRED` | на балансе проекта недостаточно средств | «Недостаточно средств на балансе проекта. Пополните баланс.» + CTA на пополнение |
| `404 NOT_FOUND` | пакет не найден или неактивен | «Пакет недоступен» + обновить список пакетов |
| `400 BAD_REQUEST / INVALID_ARGUMENT` | невалидный `pack_id` | ошибка валидации формы |
| `500 GRPC_ERROR` | сбой сервера | общий тост об ошибке, дать повторить |

Текст ошибки от бэкенда лежит в `data` (строка) — можно показать как fallback.

## 4. Состояние лимитов и пакета — экран «company state»

Главный источник состояния для UI — уже существующий эндпоинт:

```
GET /v1/pricing/company-stats
```

`data` (релевантная часть `tokens` расширена под пакеты):

```json
{
  "tokens": {
    "daily": {
      "input_tokens": 820000,
      "output_tokens": 310000,
      "plan_tokens": 1000000,
      "limit": 1000000,
      "limit_reached": true
    },
    "monthly": {
      "input_tokens": 5400000,
      "output_tokens": 2100000,
      "plan_tokens": 6200000,
      "limit": 20000000,
      "limit_reached": false
    },
    "pack_remaining": 740000,
    "active_source": "pack"
  },
  "project_count": { "current": 3, "limit": 10 },
  "builders": { "current": 1, "limit": 5 },
  "user_count": { "current": 4, "limit": 20 }
}
```

Как читать поля (это и есть «какой лимит достигнут» — то, что нужно показать):

- `daily` / `monthly`:
  - `input_tokens`, `output_tokens` — сырые суммарные токены за период (включая потраченные из пакета) — для аналитики/графиков.
  - **`plan_tokens`** — сколько потрачено именно из тарифа (без пакета). **Прогресс-бар лимита тарифа стройте по `plan_tokens` vs `limit`, а не по input+output**, иначе после покупки пакета бар «переполнится».
  - `limit` — лимит тарифа на период (`0` означает «лимит не задан / безлимит» — тогда бар не показываем).
  - `limit_reached` — достигнут ли лимит этого периода.
- `pack_remaining` — остаток пула пакета компании (дублирует `GET /token-pack/balance`).
- **`active_source`** — откуда сейчас списываются токены:
  - `"plan"` — тратится тариф, всё хорошо.
  - `"pack"` — лимит тарифа достигнут, работа идёт за счёт пакета. Показать бейдж «Работаете на токенах из пакета, осталось N».
  - `"exhausted"` — и тариф, и пакет пусты. AI-запросы будут блокироваться. Показать видный баннер «Лимит исчерпан» + CTA «Купить пакет».

## 5. Рекомендации по UX (чтобы реализовать «само и без вопросов»)

1. **Не давайте пользователю переключатель «использовать пакет».** Переключение автоматическое; UI только *отражает* состояние через `active_source`.
2. **Прогресс лимита тарифа** = `plan_tokens / limit` для дня и месяца. При `limit == 0` период считается безлимитным.
3. **Баннер апсейла:**
   - `active_source == "plan"` и оба `limit_reached == false` → ничего навязчивого.
   - один из `limit_reached == true` и `pack_remaining > 0` (`active_source == "pack"`) → мягкий бейдж «работаете на пакете, осталось N токенов», предложить докупить заранее.
   - `active_source == "exhausted"` → блокирующий баннер «Лимит исчерпан, купите пакет, чтобы продолжить».
4. **После успешной покупки** обновляйте баланс из `remaining_tokens` ответа и/или перезапрашивайте `company-stats`.
5. **Когда показывать кнопку «Купить пакет»:** всегда доступна в биллинг-разделе; агрессивно — при `limit_reached`/`exhausted`.
6. **Обработка `402`** при покупке — самый частый кейс: вести на пополнение баланса проекта, а не показывать «ошибка сервера».
7. **Единицы и валюта:** `charged_amount` и баланс проекта — в UZS. Цена пакета (`price`/`currency`) может быть в USD — показывайте в валюте пакета, но помните, что спишется эквивалент в UZS.
