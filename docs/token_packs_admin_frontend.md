# Token Packs — документация для фронтенда (админская сторона)

Этот документ описывает админскую часть «токен-пакетов»: управление каталогом пакетов — создание, редактирование, просмотр (включая неактивные) и удаление. Пользовательская часть (покупка, баланс, состояние лимитов) — в отдельном файле `token_packs_user_frontend.md`.

---

## 1. Что такое каталог пакетов и зачем им управлять

«Токен-пакет» — это карточка товара в магазине докупаемых AI-токенов. Пользователь покупает пакет, когда у его проекта закончился лимит токенов по тарифу (fare), и купленные токены автоматически подхватываются, чтобы AI-работа не вставала.

**Каталог — динамический.** Это не захардкоженный список: админ сам создаёт, редактирует и удаляет пакеты через API. Всё, что админ положит в каталог с `is_active = true`, сразу появится у пользователей в магазине.

Что важно понимать админу про логику (чтобы UI был осмысленным):

- **Пакет = название + количество токенов + цена в выбранной валюте.** Например «1M tokens» за 50000 UZS или «5M tokens» за 4 USD.
- **`product_type`** (`ugen` или `ucode`) определяет, к какому продукту относится пакет. Пользователь видит в магазине пакеты только своего продукта. Если значение не задать/прислать неизвестное — бэкенд по умолчанию проставит `ugen`.
- **Цена может быть в любой валюте** (UZS, USD и т.д.). При покупке бэкенд сам пересчитает её в UZS по курсу ЦБ и спишет с баланса проекта. То есть админ задаёт цену в удобной валюте, а конвертация — забота бэкенда.
- **Купленные токены — общий пул на уровне компании.** Это не влияет на админское управление каталогом, но объясняет, почему у пакета нет «срока годности» или «лимита в день»: это просто запас токенов.
- **Удаление — мягкое (soft-delete).** Пакет не стирается физически; он помечается удалённым и пропадает из каталога. Уже купленные пользователями токены при этом НЕ исчезают — удаляется только карточка товара из магазина, а не баланс компаний.
- **`is_active`** — это «витринный» флаг. `false` = пакет существует в каталоге, но скрыт от пользователей (черновик / временно снят с продажи). Это мягче, чем удаление: можно выключить и потом снова включить.

> Разница `is_active = false` vs удаление: первое — обратимо (снять с витрины и вернуть), второе — финально (карточка уходит из каталога). В админ-UI стоит дать обе операции и не путать их.

## 2. Аутентификация и формат ответа

Эндпоинты находятся под `/v1` и требуют те же заголовки авторизации, что и остальные вызовы gateway (`Authorization` + `project_id` в заголовках). Эти эндпоинты — административные; доступ к ним стоит давать только пользователям с админской ролью (разграничение ролей — на стороне фронта/гейтвея авторизации).

Каждый ответ обёрнут в стандартный конверт:

```json
{
  "status": "OK",
  "description": "The request has succeeded",
  "data": { /* полезная нагрузка */ },
  "custom_message": null
}
```

Полезные данные всегда в `data`. Поля с нулевыми/пустыми значениями могут отсутствовать в JSON (omitempty): считайте отсутствующее поле равным `0`/`""`/`false`.

## 3. Объект пакета (TokenPack)

Это единый объект, который возвращается из всех эндпоинтов и который вы редактируете:

```json
{
  "id": "b3f1...uuid",
  "name": "1M tokens",
  "token_amount": 1000000,
  "price": 50000,
  "currency_id": "0803582e-29d6-42fe-86ac-b4287b8fa929",
  "product_type": "ugen",
  "is_active": true,
  "created_at": "2026-06-29T10:00:00Z",
  "updated_at": "2026-06-29T10:00:00Z",
  "currency": { "id": "...", "symbol": "сум", "name": "Uzbek Som", "code": "UZS" }
}
```

| поле           | тип     | описание |
|----------------|---------|----------|
| `id`           | uuid    | идентификатор пакета (генерит бэкенд при создании; нужен для update/delete) |
| `name`         | string  | отображаемое имя пакета |
| `token_amount` | int64   | сколько токенов даёт пакет. **Должно быть > 0** |
| `price`        | number  | цена в валюте `currency_id` (поддерживает дробные, напр. `4.5` USD) |
| `currency_id`  | uuid    | id валюты цены. Можно оставить пустым, но для корректной покупки лучше задавать |
| `product_type` | string  | `ugen` или `ucode`. Пустое/неизвестное → бэкенд проставит `ugen` |
| `is_active`    | bool    | `true` — виден пользователям; `false` — скрыт (черновик/снят с продажи) |
| `created_at`   | string  | время создания (только чтение) |
| `updated_at`   | string  | время последнего изменения (только чтение) |
| `currency`     | object  | развёрнутый объект валюты (только чтение, бэкенд подтягивает по `currency_id`) |

Объект `currency` (только для отображения):

| поле     | описание |
|----------|----------|
| `id`     | id валюты |
| `symbol` | символ для отображения цены (напр. `сум`, `$`) |
| `name`   | полное название (напр. `Uzbek Som`) |
| `code`   | ISO-код (напр. `UZS`, `USD`) |

## 4. Админские эндпоинты

### 4.1. Создать пакет

```
POST /v1/token-pack
Content-Type: application/json

{
  "name": "1M tokens",
  "token_amount": 1000000,
  "price": 50000,
  "currency_id": "0803582e-29d6-42fe-86ac-b4287b8fa929",
  "product_type": "ugen",
  "is_active": true
}
```

Тело (без `id` — он генерится бэкендом):

| поле           | обязательно | примечание |
|----------------|-------------|------------|
| `name`         | да (по смыслу) | имя для витрины |
| `token_amount` | **да, > 0**    | иначе `400` |
| `price`        | да (по смыслу) | в валюте `currency_id` |
| `currency_id`  | желательно     | если пусто — пакет без привязки к валюте |
| `product_type` | нет            | пусто/неизвестно → `ugen` |
| `is_active`    | нет            | по умолчанию `false` — пакет создастся скрытым, не забудьте включить |

`data` при успехе (**HTTP 201 Created**) — полный объект `TokenPack` с проставленными `id`, `created_at`, `updated_at`, `currency`.

---

### 4.2. Редактировать пакет (полная замена)

```
PUT /v1/token-pack
Content-Type: application/json

{
  "id": "b3f1...uuid",
  "name": "1M tokens (updated)",
  "token_amount": 1000000,
  "price": 55000,
  "currency_id": "0803582e-29d6-42fe-86ac-b4287b8fa929",
  "product_type": "ugen",
  "is_active": true
}
```

**Это полная замена (full replace), а не частичное обновление.** Передавайте ВСЕ поля объекта, а не только изменённое — то, что не пришлёте, перезапишется пустым/нулевым значением. В UI: загрузите текущий объект (из списка), дайте отредактировать, отправьте целиком обратно.

- `id` — обязателен и должен быть валидным UUID (иначе `400`).
- `token_amount` должен быть `> 0` (иначе `400`).
- Если пакета с таким `id` нет — `404`.

`data` при успехе (**HTTP 200**) — обновлённый объект `TokenPack`.

> Включение/выключение витрины — это тоже PUT: меняете `is_active` и шлёте объект целиком. Отдельного эндпоинта «toggle» нет.

---

### 4.3. Удалить пакет (мягко)

```
DELETE /v1/token-pack/{id}
```

`id` — в пути, валидный UUID (иначе `400`).

При успехе — **HTTP 204 No Content** (тела нет). Пакет помечается удалённым и пропадает из всех списков (включая `only_active=false`). Уже купленные пользователями токены не затрагиваются.

> Это финальная операция для каталога. Если задача — временно снять с продажи, используйте `is_active = false` (PUT), а не удаление.

---

### 4.4. Список пакетов (включая неактивные)

```
GET /v1/token-pack?only_active=false&product_type=ugen
```

Это тот же эндпоинт, что и у пользователя, но для админки **не передавайте `only_active=true`** — иначе скрытые (`is_active=false`) пакеты не вернутся, а админу нужно видеть и их.

| параметр       | для админки |
|----------------|-------------|
| `only_active`  | `false` или не передавать — видеть ВСЕ пакеты, в т.ч. скрытые. (`true` — только активные) |
| `product_type` | опционально: фильтр `ugen` / `ucode` |

`data`:

```json
{
  "token_packs": [ { /* TokenPack */ }, { /* TokenPack */ } ],
  "count": 2
}
```

Список отсортирован бэкендом по возрастанию `token_amount`. Удалённые (soft-deleted) пакеты не возвращаются никогда.

## 5. Ошибки

| HTTP | когда | что показать админу |
|------|-------|---------------------|
| `400 BAD_REQUEST / INVALID_ARGUMENT` | `token_amount <= 0`, невалидный `id`, битый JSON | подсветить поле формы с ошибкой |
| `404 NOT_FOUND` | пакет с таким `id` не найден (update/delete) | «Пакет не найден» + обновить список |
| `500 GRPC_ERROR` | сбой сервера | общий тост, дать повторить |

Текст ошибки от бэкенда лежит в `data` (строка) — можно показать как fallback.

## 6. Рекомендации по UX для админ-панели

1. **Таблица пакетов** с колонками: имя, токены, цена + валюта, тип продукта, статус (активен/скрыт), дата создания. Источник — `GET /v1/token-pack?only_active=false`.
2. **Визуально различайте** активные и скрытые пакеты (например, приглушённая строка + бейдж «Скрыт» для `is_active=false`).
3. **Кнопка «Активен» как переключатель** удобна, но помните: под капотом это PUT с полным объектом — подгрузите текущие поля, поменяйте только `is_active`, отправьте целиком.
4. **Форма создания/редактирования** — одна и та же; в режиме редактирования предзаполняйте все поля из объекта (full replace!).
5. **Валидация на фронте до отправки:** `token_amount > 0`, `name` не пустой, цена ≥ 0, валюта выбрана. Это сэкономит лишние `400`.
6. **Цена и валюта:** дайте выбрать валюту из справочника; покажите подсказку «при покупке спишется эквивалент в UZS по курсу ЦБ».
7. **Удаление — с подтверждением** («Удалить пакет из каталога? Уже купленные токены у компаний сохранятся»). Для «временно снять с продажи» предложите вместо удаления выключить `is_active`.
8. **После любого изменения** (create/update/delete) перезапрашивайте список, чтобы таблица отражала актуальное состояние (включая пересчитанный `currency` и `updated_at`).

## 7. Краткая шпаргалка по эндпоинтам (админ)

| Действие | Метод | Путь | Тело | Успех |
|----------|-------|------|------|-------|
| Создать пакет | POST | `/v1/token-pack` | объект без `id` | 201 + TokenPack |
| Редактировать (full replace) | PUT | `/v1/token-pack` | полный объект с `id` | 200 + TokenPack |
| Удалить (soft) | DELETE | `/v1/token-pack/{id}` | — | 204 |
| Список (вкл. скрытые) | GET | `/v1/token-pack?only_active=false` | — | 200 + {token_packs, count} |
