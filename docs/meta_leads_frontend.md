# Meta (Facebook) Lead Ads — гайд для фронтенда

Этот документ описывает **всё, что нужно фронту**, чтобы построить UI интеграции Meta Lead Ads:
как подключить аккаунт, как обработать редирект (success/error), как показать страницы и формы,
как собрать маппинг полей и как показывать понятные сообщения пользователю и разработчику.

Идентификаторы, поля и JSON — на английском. Пояснения — по-русски.

---

## 1. Что это вообще такое (модель в голове)

Пользователь крутит рекламу в Facebook/Instagram с **лид-формами**. Когда человек заполняет форму,
рождается **лид**. Наша задача — поймать этот лид и положить строкой в таблицу проекта по настроенному
**маппингу полей**.

Иерархия объектов Meta:

```
Meta-аккаунт пользователя
  └── Page (Страница)            ← у страницы свой токен
        └── Lead Form (форма)    ← у формы свои questions (поля)
              └── Lead (заявка)  ← приходит к нам вебхуком
```

Чтобы всё заработало, пользователь проходит по шагам:

1. **Connect** — логинится в Meta, выдаёт нам доступ. Мы сохраняем его токен.
2. **Status** — проверяем, что токен живой (иначе просим переподключиться).
3. **Pages** — выбираем Страницу, с которой ловим лиды.
4. **Subscribe** — подписываем эту Страницу на вебхук (создаётся подключение).
5. **Forms + Questions** — у Страницы берём список форм, у формы — список полей.
6. **Mapping** — сопоставляем поля формы с полями таблицы проекта и сохраняем.
7. Готово — новые лиды автоматически падают в таблицу.

> Фронт **не общается с Facebook напрямую**. Все запросы идут на наш gateway, а он уже ходит в Meta Graph API.

---

## 2. Общие правила для всех запросов

### 2.1. База и авторизация

- Все эндпоинты ниже — на нашем gateway, префикс `/v1/facebook`.
- Авторизация — **те же заголовки, что и для любого другого запроса к gateway**:

| Заголовок | Назначение |
|---|---|
| `Authorization: Bearer <token>` | Токен пользователя платформы (не Meta!) |
| `Environment-Id: <uuid>` | Текущее окружение проекта |
| `Resource-Id: <uuid>` | Как обычно в проекте |

`project_id` и `user_id` сервер достаёт сам из токена — **их слать не нужно**.

Публичные эндпоинты (`/webhook/facebook`, `/v1/facebook/callback`) фронт **не вызывает** — их дёргает сам Facebook.

### 2.2. Единый формат ответа (envelope)

Любой ответ gateway завёрнут в один конверт:

```json
{
  "status": "OK",
  "description": "The request has succeeded",
  "data": { /* полезная нагрузка ИЛИ текст ошибки */ },
  "custom_message": null
}
```

- Полезные данные эндпоинта — всегда внутри **`data`**.
- При ошибке в `data` лежит **строка с текстом ошибки**, а `status` — машинный код (см. ниже).

### 2.3. Коды статусов

| HTTP | `status` | Когда |
|---|---|---|
| 200 | `OK` | Успех |
| 400 | `BAD_REQUEST` | Невалидное тело / бизнес-ошибка (напр. «не подключено») |
| 400 | `INVALID_ARGUMENT` | Не передан/кривой обязательный параметр |
| 500 | `GRPC_ERROR` | Ошибка внутреннего сервиса или Graph API |
| 500 | `INTERNAL_SERVER_ERROR` | Непредвиденная ошибка |
| 401 | `UNAUTHORIZED` | Проблема с авторизацией платформы |

Правило для UI: если `status !== "OK"` — показываем ошибку, текст берём из `data` (это строка).

## 3. Поток подключения (OAuth) и обработка редиректа

Это самая важная часть для UX. Разберём по шагам.

### Шаг 1. Получить ссылку авторизации

**`GET /v1/facebook/connect?redirect_uri=<полный URL текущей страницы>`**

- **`redirect_uri`** (рекомендуется явно передавать) — это URL **вашей** страницы, **куда вернуть пользователя**
  после авторизации. Мы вернём его именно туда и допишем результат в query.
  Если не передать — возьмём `Referer`, а если и его нет — `/`.

Ответ:

```json
{
  "status": "OK",
  "data": { "auth_url": "https://www.facebook.com/v21.0/dialog/oauth?client_id=...&state=...&scope=..." }
}
```

Что делает фронт: открывает `data.auth_url` — **в попапе** (рекомендуется) или редиректом всей вкладки.
Пользователь увидит окно Meta и согласится выдать доступ (scopes: управление страницами и чтение лидов).

Возможные ошибки `connect`:

| status | `data` | Что показать |
|---|---|---|
| `INVALID_ARGUMENT` | `project id is an invalid uuid` / `error getting environment id ...` | Тех. ошибка окружения, обновить страницу |
| `BAD_REQUEST` | `facebook integration is not configured` | Интеграция не настроена на бэке — сообщить админу |

### Шаг 2. Пользователь авторизуется → мы его редиректим обратно

После согласия Facebook вернёт пользователя на наш `callback`, мы обменяем код на токен, сохраним его и
**сделаем 307-редирект на ваш `redirect_uri`**, дописав результат в query-параметры:

**Успех:**
```
https://ваша-страница?facebook=success
```

**Ошибка:**
```
https://ваша-страница?facebook=error&reason=<код_стадии>&detail=<текст>
```

- `facebook` — `success` | `error` (главный флаг).
- `reason` — **на какой стадии упало** (машинный код, см. таблицу ниже) — чтобы и пользователь, и разработчик
  поняли, где именно проблема.
- `detail` — человекочитаемый текст исходной ошибки (может отсутствовать).

### Шаг 3. Фронт парсит query и реагирует

Если открывали **попапом** — страница `redirect_uri` ловит query, шлёт результат в основное окно и закрывается:

```js
// Страница, указанная в redirect_uri
const params = new URLSearchParams(window.location.search);
const fb = params.get("facebook");
if (fb) {
  window.opener?.postMessage({
    source: "facebook-oauth",
    success: fb === "success",
    reason: params.get("reason"),
    detail: params.get("detail"),
  }, window.location.origin);
  window.close();
}
```

Основное окно слушает сообщение и обновляет UI (например, перезапрашивает `GET /status`):

```js
window.addEventListener("message", (e) => {
  if (e.origin !== window.location.origin) return;
  if (e.data?.source !== "facebook-oauth") return;
  if (e.data.success) {
    toast.success("Meta подключён");
    refetchStatus();
  } else {
    toast.error(messageForReason(e.data.reason, e.data.detail));
  }
});
```

Если открывали **редиректом всей вкладки** — просто читаете query при загрузке страницы и показываете тост/баннер.

### Таблица `reason` → сообщение пользователю

| `reason` | Что произошло | Рекомендуемое сообщение |
|---|---|---|
| `facebook_access_denied` | Пользователь отклонил доступ в окне Meta | «Вы отклонили доступ. Чтобы подключить Meta, разрешите доступ к страницам.» |
| `facebook_<other>` | Любая другая ошибка от самого Meta | «Facebook вернул ошибку: <detail>. Попробуйте ещё раз.» |
| `invalid_state` | Сессия подключения истекла или ссылка недействительна (TTL 10 минут) | «Сессия подключения истекла. Начните подключение заново.» |
| `missing_code` | Facebook не вернул код авторизации | «Не получилось завершить вход в Meta. Попробуйте ещё раз.» |
| `code_exchange_failed` | Не удалось обменять код на токен | «Не удалось завершить подключение к Meta. Попробуйте позже.» |
| `long_lived_exchange_failed` | Не удалось получить долгоживущий токен | «Не удалось завершить подключение к Meta. Попробуйте позже.» |
| `token_store_failed` | Не удалось сохранить подключение | «Не удалось сохранить подключение. Попробуйте ещё раз.» |

> Любой `reason`, начинающийся с `facebook_` — это дословный код ошибки от Meta (`facebook_` + код). Для них показывайте `detail`.

## 4. Проверка статуса подключения

**`GET /v1/facebook/status`**

Дёргайте при заходе на экран интеграции и после успешного connect. Эндпоинт **всегда возвращает 200 OK**
(даже если не подключено) — смотрите на поля.

`data`:

```json
{
  "connected": true,
  "active": true,
  "expires_at": 1735689600,
  "reason": ""
}
```

| Поле | Тип | Смысл |
|---|---|---|
| `connected` | bool | Есть ли вообще сохранённое подключение Meta |
| `active` | bool | Жив ли токен прямо сейчас (проверяется у Meta) |
| `expires_at` | int64 | Unix-секунды, когда токен истекает. `0` — не истекает |
| `reason` | string | Почему `active=false` (текст для подсказки) |

Логика UI:

| connected | active | Состояние экрана |
|---|---|---|
| `false` | — | Не подключено → кнопка **«Подключить Meta»** (запускает `connect`) |
| `true` | `true` | Подключено и работает → показываем дату истечения, разрешаем работу со страницами |
| `true` | `false` | Подключение протухло → баннер **«Переподключить»** + текст из `reason` |

Подсказка: если `expires_at` близко (например, < 7 дней) — мягко предложите переподключиться заранее.

## 5. Страницы (Pages)

**`GET /v1/facebook/pages`**

Список Страниц, которыми управляет подключённый аккаунт. Пользователь выбирает одну, чтобы ловить с неё лиды.

`data`:

```json
{
  "pages": [
    { "id": "1029384756", "name": "Acme Store" },
    { "id": "5647382910", "name": "Acme Support" }
  ]
}
```

UI использует **`id`** и **`name`**. Прочие технические поля в ответе игнорируйте.

Ошибки:

| status | `data` | Показать |
|---|---|---|
| `BAD_REQUEST` | `facebook is not connected, run connect first` | Отправить на шаг подключения |
| `GRPC_ERROR` | текст ошибки Graph | «Не удалось получить страницы из Meta. Попробуйте позже.» |

---

## 6. Подписка Страницы (Subscribe)

**`POST /v1/facebook/subscribe`**

Подписывает выбранную Страницу на вебхук и создаёт подключение (`project_resource`). Идемпотентно: повторный
вызов на ту же Страницу обновит данные и **сохранит уже настроенный маппинг форм** и исходное время подключения.

Тело:

```json
{
  "page_id": "1029384756",
  "page_name": "Acme Store"
}
```

| Поле | Обязательно | Смысл |
|---|---|---|
| `page_id` | да | ID выбранной Страницы из `/pages` |
| `page_name` | нет | Кастомное имя подключения. Если пусто — берём имя Страницы |

`data`:

```json
{ "resource_id": "0e4b...uuid", "page_id": "1029384756" }
```

`resource_id` — идентификатор подключения, понадобится для отключения.

Ошибки:

| status | `data` | Показать |
|---|---|---|
| `BAD_REQUEST` | ошибка валидации (нет `page_id`) | Подсветить, что страница не выбрана |
| `BAD_REQUEST` | `facebook is not connected, run connect first` | На шаг подключения |
| `BAD_REQUEST` | `page is not managed by the connected account` | Страница больше не доступна аккаунту — обновить список `/pages` |
| `GRPC_ERROR` | текст | «Не удалось подписать страницу. Попробуйте позже.» |

## 7. Список подключений (Integration)

**`GET /v1/facebook/integration`**

Все подключённые Страницы текущего проекта/окружения вместе со статусом и сохранённым маппингом.

`data`:

```json
{
  "connected": true,
  "integrations": [
    {
      "connected": true,
      "resource_id": "0e4b...uuid",
      "page_id": "1029384756",
      "page_name": "Acme Store",
      "status": "active",
      "connected_at": "2026-06-26T08:30:00Z",
      "forms": [
        {
          "form_id": "77001",
          "form_name": "Заявка на консультацию",
          "table_slug": "leads",
          "fields": [
            { "lead_field": "full_name", "table_field": "name", "required": true },
            { "lead_field": "phone_number", "table_field": "phone", "required": true }
          ]
        }
      ]
    }
  ]
}
```

| Поле | Смысл |
|---|---|
| `connected` (верхний) | Есть ли хоть одно подключение |
| `resource_id` | ID подключения (для disconnect) |
| `page_id` / `page_name` | Какая Страница |
| `status` | `active` / `revoked` / `error` |
| `connected_at` | Когда подключили (RFC3339) |
| `forms` | Уже сохранённый маппинг (может быть пустым, если ещё не настраивали) |

Это основной источник для экрана «Мои подключения»: карточка на каждый `integrations[]`,
кнопки «Настроить маппинг» и «Отключить».

## 8. Формы и поля (для экрана маппинга)

Чтобы построить маппинг, нужно два запроса: список форм Страницы, затем поля выбранной формы.

### 8.1. Формы Страницы

**`GET /v1/facebook/pages/:page_id/forms`**

`data`:

```json
{
  "forms": [
    { "id": "77001", "name": "Заявка на консультацию", "status": "ACTIVE", "locale": "ru_RU" }
  ]
}
```

### 8.2. Поля (вопросы) формы

**`GET /v1/facebook/forms/:form_id/questions?page_id=<page_id>`**

> `page_id` обязателен в query — без него вернётся `INVALID_ARGUMENT`.

`data` (объект формы напрямую, без обёртки `forms`):

```json
{
  "id": "77001",
  "name": "Заявка на консультацию",
  "status": "ACTIVE",
  "locale": "ru_RU",
  "questions": [
    { "key": "full_name", "label": "Ваше имя", "type": "FULL_NAME" },
    { "key": "phone_number", "label": "Телефон", "type": "PHONE" },
    {
      "key": "city",
      "label": "Город",
      "type": "CUSTOM",
      "options": [ { "key": "tash", "value": "Ташкент" }, { "key": "sam", "value": "Самарканд" } ]
    }
  ]
}
```

- **`questions[].key`** — это и есть `lead_field`, левая сторона маппинга.
- `label` — человекочитаемая подпись для отрисовки в UI.
- `type` — тип поля Meta (`FULL_NAME`, `PHONE`, `EMAIL`, `CUSTOM`, ...). Можно использовать для иконок/валидации.
- `options` — для полей с выбором.

Ошибки 8.1/8.2:

| status | `data` | Показать |
|---|---|---|
| `INVALID_ARGUMENT` | `page_id is required` / `form_id is required` / `page_id query param is required` | Тех. ошибка, не должно случаться при корректном UI |
| `BAD_REQUEST` | `page ... is not managed by the connected account` | Обновить список страниц |
| `GRPC_ERROR` | текст | «Не удалось получить формы/поля из Meta.» |

## 9. Сохранение маппинга (Mapping)

**`PUT /v1/facebook/mapping`**

Сохраняет соответствие «поле формы → поле таблицы». Формы, которых нет в этом запросе, при приёме лида
**пропускаются**. То есть отправляйте полный набор форм, которые хотите принимать.

Тело:

```json
{
  "page_id": "1029384756",
  "forms": [
    {
      "form_id": "77001",
      "form_name": "Заявка на консультацию",
      "table_slug": "leads",
      "fields": [
        { "lead_field": "full_name",   "table_field": "name",  "required": true },
        { "lead_field": "phone_number","table_field": "phone", "required": true },
        { "lead_field": "city",        "table_field": "city",  "required": false }
      ]
    }
  ]
}
```

| Поле | Обязательно | Смысл |
|---|---|---|
| `page_id` | да | Какая Страница (должна быть подписана через subscribe) |
| `forms[].form_id` | да | ID формы из `/forms` |
| `forms[].form_name` | нет | Для отображения |
| `forms[].table_slug` | да | Slug целевой таблицы проекта |
| `forms[].fields[].lead_field` | да | `key` вопроса из `/questions` |
| `forms[].fields[].table_field` | да | Поле в таблице проекта |
| `forms[].fields[].required` | нет | Если `true` и значение пустое — лид по этой форме не запишется |

`data`:

```json
{ "resource_id": "0e4b...uuid" }
```

Ошибки:

| status | `data` | Показать |
|---|---|---|
| `BAD_REQUEST` | ошибка валидации (нет обязательного поля) | Подсветить незаполненные поля маппинга |
| `BAD_REQUEST` | `facebook page is not connected for this project` | Сначала сделать subscribe этой Страницы |
| `GRPC_ERROR` | текст | «Не удалось сохранить маппинг. Попробуйте позже.» |

### Как собрать UI маппинга

1. Берём `table_slug` → тянем поля таблицы проекта (вашим существующим API схемы таблиц).
2. Слева — `questions` формы (`label`, ключ `key`), справа — селект полей таблицы.
3. Чекбокс `required` на строку.
4. По «Сохранить» — собираем тело и шлём `PUT /mapping`.

## 10. Отключение (Disconnect)

**`DELETE /v1/facebook/integration/:id`**

`:id` — это `resource_id` из `/integration` или из ответа `subscribe`. Отписывает Страницу от вебхука и удаляет
подключение.

`data`:

```json
{ "resource_id": "0e4b...uuid" }
```

Ошибки:

| status | `data` | Показать |
|---|---|---|
| `INVALID_ARGUMENT` | `resource id is required` | Тех. ошибка |
| `GRPC_ERROR` | текст | «Не удалось отключить. Попробуйте позже.» |

UI: спросить подтверждение («Лиды с этой страницы перестанут поступать»), затем рефетч `/integration`.

## 11. Рекомендуемые экраны (сводка)

1. **Экран интеграции Meta**
   - Запрос `GET /status` при входе.
   - Не подключено → кнопка «Подключить Meta» (`connect` → попап → обработка редиректа).
   - Подключено, токен протух → баннер «Переподключить» (по `reason`).
   - Подключено и активно → блок «Мои подключения».

2. **Мои подключения** (`GET /integration`)
   - Карточка на каждую Страницу: имя, статус, дата, число настроенных форм.
   - Кнопка «Добавить страницу» → выбор из `GET /pages` → `POST /subscribe`.
   - Кнопки «Маппинг» и «Отключить».

3. **Мастер маппинга**
   - `GET /pages/:id/forms` → выбрать формы.
   - `GET /forms/:id/questions?page_id=` → построить таблицу соответствий.
   - `PUT /mapping` → сохранить.

## 12. Справочник эндпоинтов

| Метод | Путь | Назначение |
|---|---|---|
| GET | `/v1/facebook/connect?redirect_uri=` | Получить `auth_url` для входа в Meta |
| GET | `/v1/facebook/status` | Статус токена: connected / active / expires_at |
| GET | `/v1/facebook/pages` | Страницы аккаунта |
| GET | `/v1/facebook/pages/:page_id/forms` | Формы Страницы |
| GET | `/v1/facebook/forms/:form_id/questions?page_id=` | Поля формы |
| POST | `/v1/facebook/subscribe` | Подписать Страницу, создать подключение |
| GET | `/v1/facebook/integration` | Список подключений + маппинги |
| PUT | `/v1/facebook/mapping` | Сохранить маппинг форм |
| DELETE | `/v1/facebook/integration/:id` | Отключить Страницу |

Публичные (Facebook → backend, фронт не трогает): `GET/POST /webhook/facebook`, `GET /v1/facebook/callback`.

## 13. Приложение: как лид попадает в таблицу (контекст)

Фронт здесь ничего не делает — для понимания общей картины:

1. Человек заполняет лид-форму в Facebook/Instagram.
2. Meta шлёт на наш вебхук `page_id + form_id + leadgen_id`.
3. Backend по `page_id` находит подключение(я), по `form_id` — маппинг.
4. По токену Страницы тянет данные лида из Graph API.
5. Пишет строку в `table_slug` по маппингу. Дубликаты вебхука защищены детерминированным `guid` от `leadgen_id`.

То есть после успешного `subscribe` + `mapping` лиды появляются в таблице автоматически, без участия UI.
