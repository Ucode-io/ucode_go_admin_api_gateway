# Ugen Templates — ФРОНТ (пользователь): платное добавление пользователей (per-user seat)

Эта дока описывает, что меняется на фронте, когда проект создан из шаблона с **ценой за пользователя**
(`per_user_price`). Коротко: **добавление каждого пользователя в такой проект — платное**, деньги
списываются с баланса head-проекта. У обычных проектов ничего не меняется.

> Связанные доки: `ugen_template_pricing_frontend.md` (использование шаблона + разовая оплата импорта),
> `ugen_template_per_user_pricing_admin.md` (как админ задаёт цену за юзера).

---

## 1. Суть фичи (в двух абзацах)

Админ может задать шаблону **цену за пользователя** (`per_user_price`). Когда пользователь создаёт проект из
такого шаблона, эта цена **копируется в проект**. После этого **каждое добавление нового пользователя** в
проект (создание нового или приглашение существующего) **списывает `per_user_price`** с баланса head-проекта.

Пока у проекта `per_user_price > 0`, **тарифный лимит на количество юзеров игнорируется** — можно добавлять
сколько угодно, но каждый платный. Если `per_user_price = 0` (обычные проекты и проекты из бесплатных
шаблонов) — всё работает как раньше: действует лимит юзеров по тарифу, без списаний.

---

## 2. Как фронту понять, что проект «платный за юзеров»

В объекте **проекта** появились два новых поля:

| Поле | Тип | Значение |
|------|-----|----------|
| `per_user_price` | number | Цена за одного добавляемого юзера. `0` = бесплатно (лимит по тарифу). |
| `per_user_currency_id` | string (uuid) | Валюта цены. Пусто (`""`) = **UZS**. |

Правило для UI:
- `per_user_price > 0` → проект **платный за юзеров**. Показывайте цену рядом с кнопкой «Добавить
  пользователя» и стройте флоу с подтверждением оплаты (см. §5).
- `per_user_price === 0` → обычный флоу добавления, без упоминания оплаты.

> Эти поля возвращаются там же, где вы уже получаете данные проекта (объект `Project`). Подтяните их при
> загрузке экрана управления пользователями проекта.

## 3. Общие правила API

### Формат ответа (обёртка)
```json
{
  "status": "OK",
  "description": "The request has succeeded",
  "data": { ... },
  "custom_message": ""
}
```

### Авторизация
Стандартные заголовки (те же, что вы уже используете для управления пользователями):
```
Authorization: Bearer <token>
Environment-Id: <environment_id>
Resource-Id: <resource_id>
```

### HTTP-коды
| Код | status | Когда |
|-----|--------|-------|
| 200 / 201 | `OK` / `Created` | Пользователь добавлен (и, если платно, деньги списаны) |
| 400 | `BAD_REQUEST` / `INVALID_ARGUMENT` | Кривое тело / невалидные id |
| 401 | `UNAUTHORIZED` | Нет/невалидный токен |
| 402 | `PAYMENT_REQUIRED` | **Недостаточно средств** (платный юзер) **или** достигнут тарифный лимит (бесплатный проект) |
| 409 / 500 | — | Пользователь уже в проекте / внутренняя ошибка |

## 4. Эндпоинты добавления пользователя

> Это **существующие** эндпоинты управления пользователями. Их сигнатура **не поменялась** — изменилось
> только поведение: для проектов с `per_user_price > 0` они теперь **списывают деньги** и **не проверяют
> тарифный лимит**. Фронту не нужно ничего слать по-новому — нужно лишь корректно обрабатывать `402`.

### 4.1. Создать нового пользователя и добавить в проект
```
POST /v2/user
```
Тело (ключевые поля):
```json
{
  "login": "ivan",
  "password": "secret",
  "email": "ivan@example.com",
  "phone": "+998901234567",
  "name": "Иван",
  "project_id": "uuid",          // обязательно
  "role_id": "uuid",             // обязательно
  "client_type_id": "uuid",      // обязательно
  "client_platform_id": "uuid",
  "additional_data": { }
}
```
- **200/201:** `data` = созданный пользователь.
- Если проект платный за юзеров — перед созданием **списывается `per_user_price`** с head-баланса.

### 4.2. Пригласить существующего пользователя в проект
```
POST /v2/user/invite
```
Тело:
```json
{
  "user_id": "uuid",             // обязательно
  "project_id": "uuid",          // обязательно
  "role_id": "uuid",             // обязательно
  "client_type_id": "uuid",      // обязательно
  "env_id": "uuid",
  "additional_data": { }
}
```
- **200:** `data` = результат добавления.
- Если проект платный за юзеров — **списывается `per_user_price`**.

### Что за что платит
| Ситуация | Списание |
|----------|----------|
| **Первый** пользователь проекта | ❌ **Бесплатно** (см. §4.1) |
| Добавлен **новый** пользователь (2-й и далее, в платном проекте) | ✅ Списывается `per_user_price` |
| Приглашён **существующий** пользователь, которого ещё нет в проекте (2-й и далее) | ✅ Списывается `per_user_price` |
| Пользователь **уже состоит** в проекте | ❌ Не списывается — вернётся ошибка «уже участник» |
| Проект с `per_user_price = 0` | ❌ Не списывается — действует тарифный лимит |

#### 4.1. Первый пользователь — всегда бесплатный
У каждого проекта **первый** пользователь добавляется **бесплатно**, даже если `per_user_price > 0`. Это
тот самый владелец/owner, который создаётся сразу после импорта шаблона: при «использовать шаблон»
списывается только разовая цена импорта (`price`), а не `price + per_user_price`. Плата за юзера начинается
**со второго** пользователя.

> Для UI: если в проекте ещё **нет** пользователей — не показывайте ценник на кнопке добавления и не стройте
> подтверждение оплаты (первый юзер бесплатный). Со второго — показывайте цену.

## 5. Обработка `402` — два разных случая

`402` может прийти по **двум** причинам. Различайте их по `data.code`:

### 5.1. Недостаточно средств (платный за юзеров проект)
```json
{
  "status": "PAYMENT_REQUIRED",
  "data": {
    "type": "payment_required",
    "code": "insufficient_balance"
  }
}
```
→ На балансе head-проекта не хватает денег на `per_user_price`. Пользователь **не добавлен**, деньги **не
списаны**. Покажите модалку «Недостаточно средств» и кнопку пополнения баланса.

### 5.2. Достигнут тарифный лимит (обычный/бесплатный проект)
```json
{
  "status": "PAYMENT_REQUIRED",
  "data": {
    "type": "payment_required",
    "code": "user_limit",        // или "builders_limit"
    "unit": "users"              // или "builders"
  }
}
```
→ Это старое поведение для проектов **без** цены за юзера. Покажите «Достигнут лимит пользователей по
тарифу» и предложите апгрейд тарифа.

> Правило: `data.code === "insufficient_balance"` → пополнить баланс; `data.code === "user_limit" |
> "builders_limit"` → апгрейд тарифа. (В некоторых ответах для лимита может прийти только
> `type: "payment_required"` без `code` — трактуйте это как лимит тарифа.)

---

## 6. Гарантии и поведение сервера (что важно знать фронту)

1. **Атомарность:** для платного проекта сначала проверяется/списывается баланс, только потом создаётся
   пользователь. Не хватило денег → `402`, юзер не создан.
2. **Авто-возврат:** если списание прошло, но добавление пользователя упало — сервер **автоматически
   возвращает** деньги. Фронту ничего откатывать не нужно; можно показать «Не удалось, попробуйте ещё раз».
3. **Без двойного списания:** повторное добавление одного и того же юзера в тот же проект **не** спишет
   деньги повторно — вернётся ошибка «уже участник» (`409`/ошибка). Ретрай после таймаута безопасен: если
   юзер уже добавился, повтор вернёт «уже участник», а не второе списание.
4. **Валюта и конвертация:** `per_user_price` списывается в валюте проекта, сервер сам конвертирует в UZS по
   курсу. Фронту достаточно показать `per_user_price` + символ валюты по `per_user_currency_id`.

## 7. 💎 Рекомендованный UX/UI

### 7.1. Экран «Пользователи проекта»
Показывайте оплату только когда в проекте **уже есть** пользователь (первый — бесплатный, §4.1).
Если `project.per_user_price > 0` **и** в проекте есть хотя бы 1 пользователь, покажите информ-баннер сверху:
```
┌───────────────────────────────────────────────────────────┐
│ ⓘ В этом проекте добавление пользователя платное:          │
│   5 000 UZS за каждого. Списывается с баланса проекта.     │
└───────────────────────────────────────────────────────────┘
```
Кнопка добавления — с ценником:
```
[ + Добавить пользователя · 5 000 UZS ]
```

### 7.2. Модалка подтверждения (перед платным добавлением)
```
┌─ Добавить пользователя ───────────────────────────┐
│ Иван (ivan@example.com)                           │
│                                                   │
│ Стоимость: 5 000 UZS  (спишется с баланса проекта)│
│ Баланс проекта: 42 000 UZS → станет 37 000 UZS    │
│                                                   │
│        [ Отмена ]     [ Добавить за 5 000 UZS ]   │
└───────────────────────────────────────────────────┘
```
- Показывайте текущий баланс и остаток после списания (баланс берите из объекта head-проекта — поле
  `balance`).
- Если баланса заведомо не хватает — блокируйте кнопку и сразу предлагайте пополнить (не дожидаясь `402`).

### 7.3. Состояния и обработка ответов
| Ответ | Действие UI |
|-------|-------------|
| 200/201 | Тост «Пользователь добавлен, списано 5 000 UZS». Обновить список и баланс. |
| 402 `insufficient_balance` | Модалка «Недостаточно средств» + кнопка «Пополнить баланс». Юзер не добавлен. |
| 402 `user_limit`/`builders_limit` | «Достигнут лимит тарифа» + «Улучшить тариф». |
| 409 / «уже участник» | Тост «Пользователь уже в проекте». Без списания. |
| 400 | Показать ошибки валидации формы. |
| 500 | «Не удалось добавить, попробуйте ещё раз». Если было списание — деньги уже вернулись. |

### 7.4. Бесплатный проект (`per_user_price === 0`)
Никаких ценников и подтверждений оплаты — обычный флоу добавления. `402` в этом случае означает лимит
тарифа (см. §5.2), а не деньги.
