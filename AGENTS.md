# Stack
- Go + Gin + gRPC
- PostgreSQL (pgx/v5), Redis
- Git/GitHub, CI/CD

# Behaviour
- Общение: русский
- Уровень: middle — объясняй нетривиальные решения в 1-2 строки, очевидное пропускай
- Без вступлений, похвалы и повторения задачи
- Если задача ясна — сразу код

# Code Output
- Показывай ТОЛЬКО изменённые функции/блоки
- Перед блоком указывай файл: `// internal/handler/user.go`
- Комментарии только где логика неочевидна
- Если меняешь поведение — одной строкой объясни почему

# Go Rules
- context.Context — всегда первым аргументом
- Никаких глобальных переменных и init()
- Интерфейсы определяй на стороне потребителя
- Структуры инициализируй с именами полей (не positional)

# PostgreSQL
- Только pgx/v5, не database/sql
- Транзакции: defer tx.Rollback() + явный tx.Commit()
- Избегай N+1 — спроси если неясно как батчить

# Redis
- pipelining где уместно
- Явно указывай TTL, никогда 0 (без expire)

# gRPC
- Валидируй proto-поля в начале handler'а
- Возвращай status.Error(codes.X, "...") — не errors.New
- Interceptors для auth/logging/recovery

# Clarification
- Если задача неясна или неоднозначна — задай ОДИН уточняющий вопрос, не угадывай
- Не додумывай детали самостоятельно — лучше спросить чем переделывать
- Если вариантов решения несколько и выбор неочевиден — спроси какой предпочесть

# Architecture
- Один файл = одна ответственность: middleware, worker, handler, helper — в отдельных файлах
- Пакеты: без underscores (`apilimits`, `chatprompts`), имя пакета = имя директории без спецсимволов
- Прежде чем добавлять функцию в существующий файл — проверь, не нужен ли отдельный файл
- Embedded struct вместо копипаста lifecycle (Start/Stop, wg, cancelFn)
- Константы выноси в файл с константами пакета, не разбрасывай по файлам
- Если вижу архитектурную проблему (дублирование, неверный пакет, god-file) — говорю сразу, исправляю вместе с задачей
- Экспортируй тип один раз из его canonical пакета; не дублируй struct в разных пакетах
- Не вали всё в `v1/`: billing, ai, mcp, github — каждый в свой subpackage
- Structs/types: в том же файле только если используется в одном месте и не экспортируется; если используется в 2+ файлах или это публичный request/response/entity — выноси в `api/models/` в тематический файл (mcp_models.go, ai_chat_models.go и т.д.)

# Code Quality (non-negotiable)
- Мёртвый код: сначала спроси "эти функции больше не нужны, удалить?" — только после подтверждения удаляй
- Никакого копипаста — если логика повторяется 2+ раза, выноси в helper
- Каждая функция делает одно — если название содержит "and/и", это два метода
- Не пиши defensive code для ситуаций которые не могут произойти
- После любого изменения — код должен компилироваться (go build ./...)

# Token Saving (for Codex)
- Не предлагай альтернативы если не просят
- Не пиши "можно также сделать..."
- Один вопрос за раз если нужно уточнить
- /compact после каждой завершённой задачи
